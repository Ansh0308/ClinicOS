ClinicOS — Clinic Management & Patient Experience Platform

Version 3.0 Final Confidential 26 February 2026 20 Sections · 25 Features

1. Project Overview & Goals

1.1 Executive Summary

ClinicOS is an all-in-one clinic management platform that digitizes the complete patient lifecycle — from appointment booking and queue management to billing, visit history, and automated multi-channel communication. It eliminates paper-based processes, reduces wait times, and gives every role in a clinic (patient, doctor, staff, admin) a tailored, fast, and intuitive interface.

1.2 Problem Statement

  • Patients wait 45–90 minutes with zero visibility into queue status or realistic ETAs
  • Doctors cannot recall last visit details, medications, or complaints without paper files
  • Front-desk staff manually manage tokens on whiteboards, leading to errors and queue chaos
  • Bills are printed on paper — no digital trail, no audit history, no easy resending
  • No automated communication; patients miss turns or arrive far too early
  • Clinic owners have no real-time visibility into revenue, throughput, or wait performance

1.3 MVP Definition

A working web app where staff can register/look up patients by phone, create and serve tokens, doctors can view history and record consultations, and the system sends automated WhatsApp/SMS/Email messages and records bills and visit history — accessible via role-based dashboards.

1.4 Strategic Goals

  • Reduce front-desk check-in to < 20 seconds for returning patients
  • Replace manual token boards with automated real-time queue with ETA
  • Give doctors full patient history in < 5 seconds
  • Automate 100% of routine patient communications (WhatsApp / SMS / Email)
  • Automate billing receipt delivery for all paid bills
  • Provide clinic owners live financial and operational dashboards
  • Achieve 40% reduction in average patient wait time within 3 months

5. Must-Have Features (F1–F7)

All P0 features required in Phase 1 MVP. Each includes description, interaction flow, technical requirements, and success criteria.

F1 — Smart Patient Entry: Phone Lookup & Quick Registration

Receptionist types a phone number → instant lookup. Returning patients auto-fill in <1s. New patients get a minimal quick-create form. The entire check-in experience hinges on this feature.

Technical Requirements

  • Indexed DB search on phone field; API POST /api/patients/lookup returns match in ≤500ms
  • Redis caching for recent lookups (last 500 phone lookups per clinic per day)
  • Phone normalization: strip country codes, whitespace, dashes before lookup
  • Fuzzy match for common formatting variations; unique constraint on normalized phone + clinic_id
  • Idempotent create path: upsert pattern prevents duplicate creation on double-submit
  • Consent capture at creation: opt_in_messaging boolean + consent_timestamp + source

Success Criteria

  • ≥90% of returning patients auto-identified within 1 second of phone entry
  • Lookup API P50 <500ms; P99 <1 second under load
  • Time from phone entry to token issued: <30s for new patient; <20s for returning
  • Zero duplicate patient records under concurrent registration attempts

F2 — Queue & Token Management: Live Line Tracking

Tokens created for walk-ins and appointments, displayed on an interactive real-time queue board visible across all connected devices. Supports multi-doctor queues and service-based queues.

Technical Requirements

  • WebSocket (Socket.IO) for real-time sync — queue changes broadcast within 1 second
  • Persistent queue state in PostgreSQL + Redis hot cache
  • Optimistic locking + DB transactions for reorder and status changes
  • ETA engine: moving average of last 10 consult durations per doctor; configurable floor/ceiling
  • Audit log entry for every token status change and reorder
  • Multi-tenant: queues fully isolated per clinic

F3 — Messaging Automation: WhatsApp / SMS / Email Triggers

Fully configurable templates with event-driven triggers. Core triggers: token issued, two-patients-before reminder, your-turn, queue pause/resume, bill sent, appointment confirmed/reminder, lab return.

Technical Requirements

  • Integration with Meta WhatsApp Business API or Twilio; abstraction layer for provider switch
  • Message broker (RabbitMQ or AWS SQS) with exponential back-off retry (3 attempts) and DLQ
  • Provider webhook endpoint to receive delivery receipts and update MessageLog
  • Template engine: variable substitution, conditional blocks, Unicode support
  • Throttling and cost controls: daily message cap per patient (configurable)
  • SMS DLT registration (India): all SMS templates pre-registered

Success Criteria

  • Message send success rate ≥95% across provider delivery reports
  • Trigger-to-send latency P50 <10 seconds; P95 <30 seconds
  • 100% of sends respect opt-out (zero messages sent to opted-out patients)

F4 — Visit / EHR Persistence: Doctor Remembers Everything

Complete structured EHR for each consultation: complaint, vitals, diagnosis, investigations, prescriptions with dosage details, follow-up instructions, file attachments, and billing linkage. Fully auditable.

Technical Requirements

  • Normalized data model: Visit → Prescription[], TestOrder[], Attachment[]
  • WYSIWYG/structured form with autosave every 30 seconds; versioned edits
  • Attachments stored in S3-compatible storage (encrypted); access via pre-signed URLs with 24h expiry
  • Audit trail: every save records (user_id, timestamp, JSON diff)
  • Full-text search across complaint, diagnosis, and notes fields (Postgres tsvector)
  • PDF export: printable prescription with clinic letterhead, doctor signature, QR to digital record

Success Criteria

  • Doctor retrieves full patient history within ≤5 seconds for 95% of requests
  • 100% of visits saved with complete audit metadata
  • Zero data loss on autosave (validated by disconnection test scenarios)

F5 — One-Tap Operational Controls: Pause, Lab, Emergency, Bring-back

Single-click controls for managing operational irregularities: pausing the queue, sending a patient for lab tests, bringing them back with priority, and emergency push-to-front. Each action is atomic, auditable, and can trigger a patient notification.

F6 — Billing & Digital Receipts

Create and manage itemized bills linked to a visit. Supports configurable tax rules (GST), discounts, multiple payment modes, partial payments, refunds, and generates a branded PDF receipt. Receipts sent automatically via messaging engine.

Success Criteria

  • Bill creation (open form → PDF generated): <2 minutes end-to-end
  • Receipts sent for 100% of bills where patient has opted in for messaging
  • PDF generation ≤5 seconds for standard bills (<20 line items)
  • Payment reconciliation accuracy 100% (invoice total = sum of payments recorded)

F7 — Clinic Statistics & Analytics Dashboard

Real-time and historical KPIs: patients today, revenue, average wait time, token throughput, no-shows, top complaints/services, doctor utilisation, and message delivery stats. Interactive visualizations with date/doctor filters, drilldowns, and scheduled exports.

6. Should-Have Features (F8–F13)

F8 — Appointment Booking & Calendar Integration

Patients book appointments online by selecting service, doctor, and available time slot. System prevents double-booking, sends confirmation and reminder notifications, and optionally syncs to external calendars (Google / Outlook). Slot holds via Redis TTL (5 minutes).

F9 — Messaging Template Editor & Consent Management

Admin and staff can create, edit, test, and version message templates for all channels. Consent management provides full view of patient opt-in/opt-out status per channel, with bulk tools fully integrated with messaging engine.

F10 — Global Search & Patient Directory

Global search across patients, visits, bills, tokens, and appointments. Supports searching by phone, name, token number, bill ID, and free-text in notes (with PHI access controls). Search latency P50 <300ms; P99 <800ms.

F11 — Audit Logs & Compliance

Immutable append-only log of all critical system actions. Supports compliance exports and forensic investigation. 100% of defined critical actions produce an audit entry (validated by integration tests).

F12 — Reports & Financial Exports

Generate financial, operational, and compliance reports with date/doctor/service filters. Viewable in-app and exportable as PDF or CSV. Scheduled report delivery by email on weekly or monthly cadence.

F13 — Role-Based Access Control (RBAC)

Comprehensive permission system enforced at API layer and UI layer. RBAC matrix: role × resource × action (create/read/update/delete/admin). JWT claims carry role; role changes invalidate existing JWT within 5 minutes via Redis-backed token blocklist.

7. Additional Features (F14–F19)

F14 — Payments & Refunds Integration

Online payments for appointment booking and bill settlement via Razorpay / Stripe. Handles webhook confirmation, generates receipts, supports partial payments and refunds. PCI-DSS compliance via gateway tokenization.

F15 — Lab Orders & Results Integration

Create lab orders from consultation UI, track sample status, and attach results to the patient's visit record. Supports integrated labs (HL7/FHIR API) and manual result upload for non-integrated labs.

F16 — Multi-clinic / Multi-location Support

Single admin account manages multiple clinic branches with shared staff (optional), per-location reporting, and centralized billing. Each branch has its own queue, patients, and services while sharing global configuration.

F17 — Offline Resilience & Background Sync

Critical operations continue briefly during poor connectivity: patient lookup from cache, new token creation (queued locally via IndexedDB), and visit note drafting. Background sync resumes once connectivity returns with conflict resolution.

F18 — Kiosk Mode & In-Clinic Token Display

Public-facing display (TV or tablet) shows current token being served, next tokens, estimated wait times, and clinic notice board. Optionally supports self-service token printing for walk-in patients.

F19 — Telemedicine / Video Consult

Host secure video consultations integrated into the doctor workflow. WebRTC via Twilio Video SDK or self-hosted Jitsi. EHR and billing flows remain identical to in-person visits.

8. Additional Features — New Additions (F20–F25)

Six features absent from reference PRDs but considered important for a complete, production-grade clinic platform.

F20 — Prescription Template Library (Doctor Quick-Rx)

Doctors save common prescriptions as named templates. During consultation, select a template to instantly populate the entire prescription block. Dramatically reduces consultation time for routine cases.

F21 — Patient QR Self-Check-in

Returning patients scan a QR code at clinic entrance to check in and join queue without waiting for staff. QR encodes a short-lived signed URL (clinic_id + date + HMAC signature; 24h validity). Anti-abuse: rate-limit per IP (max 10 check-ins per IP per hour).

F22 — Staff Shift Handover & Clinic Daily Notes

Structured daily shift log allows staff to record important notes at start and end of each shift — pending tasks, patient escalations, cash balance, equipment issues. Prevents information loss between shifts.

F23 — Patient Referral Tracking

Doctors can refer a patient to another doctor or specialist. Referral recorded in patient's visit, receiving doctor notified, and outcome logged when patient returns. Closes the clinical loop and builds a richer patient record.

F24 — Two-Factor Authentication (2FA)

All staff, doctor, and admin accounts require 2FA. TOTP (RFC 6238) support as alternative to SMS OTP. OTP stored as HMAC hash (not plaintext); expires after 5 minutes; single-use. Brute force protection: OTP locked after 5 failed attempts; 15-minute cooldown.

F25 — API Rate Limiting & Abuse Prevention

Protect all API endpoints against abuse, brute-force attacks, and bot traffic. Rate limiting via Redis sliding window counters per IP, per user, and per API key. DDoS mitigation at infrastructure level (Cloudflare WAF or AWS Shield).

12. Privacy & Messaging Consent

Consent Capture Flow

  • Staff-side registration: consent checkbox shown (default: checked). Staff must explicitly uncheck to deny.
  • Patient portal signup: explicit checkbox for appointment and queue notifications.
  • Self-check-in QR: consent checkbox shown before token creation.
  • Consent version tracked: if privacy policy version changes, re-consent prompt triggered.

Data Privacy Requirements

  • PHI encrypted at rest: AES-256 for PostgreSQL column-level encryption on sensitive fields
  • Data in transit: TLS 1.3 minimum on all endpoints; HSTS with 1-year max-age; no HTTP fallback
  • Data retention: visit records ≥7 years; billing records ≥5 years; audit logs ≥5 years
  • Soft delete only: no hard deletion of patient, visit, or billing records
  • DPDP Act 2023 (India) compliance: explicit consent, purpose limitation, data minimization, right to correction

3. Backend — Node.js 20 LTS + Express.js

Why Node.js is the Right Backend for ClinicOS

ClinicOS's most demanding backend feature is the real-time queue: every token status change must broadcast to all connected clients within 1 second. Node.js's event-loop architecture is built for exactly this — handling many concurrent WebSocket connections efficiently without spawning OS threads per connection.

PRD RequirementWhy Node.js Handles It Well
Real-time queue updates ≤1s broadcastSocket.IO runs natively in Node's event loop; no thread context-switching overhead; pub/sub via Redis adapter scales across multiple Node instances
200 concurrent staff/doctor sessionsNode handles 10,000+ concurrent connections on a single process; much cheaper than multi-threaded PHP/Java at this scale
API response P95 <500ms on phone lookupRedis-cached lookups are I/O-bound (not CPU-bound); Node's async I/O maximises Redis throughput
PDF generation (Puppeteer)Puppeteer runs as a child process; Node spawns it async and responds when ready without blocking the event loop
Multi-tenant clinic isolationMiddleware can extract clinic_id from JWT and scope all DB queries in one centralized place

4. Database — PostgreSQL 16

PostgreSQL is strongly recommended over MySQL for ClinicOS. The specific reasons are explained below with direct PRD references.

PRD FeaturePostgreSQL 16 ✅MySQL 8.x ⚠️
Prescription storage (JSONB)JSONB — binary JSON with indexing; query inside JSON without fetching full row. Used in Visit.prescriptions, Clinic.settings, Bill.itemsJSON type supported but no native GIN/BRIN indexes on JSON; slower range queries
Full-text search (PRD F10)Built-in tsvector full-text search across complaint, diagnosis, notes. No extra service for MVPFull-text is CHAR-based and less expressive; often requires Elasticsearch sooner
Row-level security (PRD §12)RLS policies enforce patient data isolation at DB level — not just application codeNo native RLS; security must be enforced purely in application code (higher breach risk)
Materialized views (PRD F7 analytics)Native REFRESH MATERIALIZED VIEW CONCURRENTLY — analytics views refresh without locking readsNo native materialized views; must simulate with stored procedures
Array fields (complaint_tags[])Native array columns with GIN index — complaint_tags TEXT[] searchable efficientlyNo native arrays; requires junction table for tags (extra join, extra migration)

Key Schema (PostgreSQL-specific)

-- Patient table (multi-tenant, consent tracked)
CREATE TABLE patients (
  patient_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  clinic_id  UUID NOT NULL REFERENCES clinics(clinic_id),
  phone      VARCHAR(15) NOT NULL,
  name       VARCHAR(200),
  dob        DATE,
  opt_in_messaging BOOLEAN DEFAULT true,
  consent_timestamp TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(clinic_id, phone)
);

-- Visit with JSONB prescription (no extra table needed)
CREATE TABLE visits (
  visit_id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  patient_id     UUID REFERENCES patients(patient_id),
  doctor_id      UUID REFERENCES doctors(doctor_id),
  clinic_id      UUID NOT NULL,
  complaint      TEXT,
  complaint_tags TEXT[],           -- Native array, GIN indexed
  diagnosis      TEXT,
  notes          TEXT,             -- Doctor-private notes
  prescriptions  JSONB,            -- [{name, dose, freq, duration}]
  tests_ordered  TEXT[],
  follow_up_date DATE,
  created_at     TIMESTAMPTZ DEFAULT NOW()
);

-- Full-text search index
ALTER TABLE visits ADD COLUMN search_vector TSVECTOR
  GENERATED ALWAYS AS (
    to_tsvector('english', coalesce(complaint,'') || ' ' ||
    coalesce(diagnosis,'') || ' ' || coalesce(notes,''))
  ) STORED;
CREATE INDEX idx_visits_search ON visits USING GIN (search_vector);

-- Token/Queue with status ENUM
CREATE TYPE token_status AS ENUM
  ('waiting','now','on_hold','served','cancelled');

CREATE TABLE tokens (
  token_id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  clinic_id         UUID NOT NULL,
  patient_id        UUID REFERENCES patients(patient_id),
  token_number      INT NOT NULL,
  queue_position    INT,
  status            token_status DEFAULT 'waiting',
  estimated_wait_mins INT,
  issued_at         TIMESTAMPTZ DEFAULT NOW(),
  called_at         TIMESTAMPTZ,
  served_at         TIMESTAMPTZ
);
